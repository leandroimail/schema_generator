# Schema Matching Project

## Overview

This project provides a comprehensive framework for schema matching, data profiling, and dictionary generation using advanced LLMs (Large Language Models) and data science techniques. It is specifically designed to automate the integration and mapping of data from normalized relational models (such as those in 3rd Normal Form) to multidimensional star schemas, which are widely used in analytical and data warehouse environments.

The core of the project is a prompt-driven approach: LLMs are guided by detailed instructions (see `prompts/schema_match_generation.md`) to analyze, compare, and map fields between source (relational) and target (star schema) models. The process includes semantic analysis, detection of necessary data transformations (such as aggregations, type conversions, and concatenations), and ensures that all target fields are covered—even when no direct match exists. This enables the automation of complex ETL and data integration tasks that are typically manual and error-prone.

The project leverages configuration files, data dictionaries, profiling outputs, and a set of modular Python scripts to automate and streamline the schema matching process. It supports multiple LLM providers (OpenAI, Google Gemini, DeepSeek) and is highly configurable via YAML and environment files.

---

## Motivation

Integrating data from different sources—especially when moving from transactional, normalized databases to analytical, multidimensional models—requires aligning schemas that may use different names, types, or structures for similar concepts. Manual schema matching in this context is particularly challenging due to:

- The need to map multiple normalized tables and fields to denormalized fact and dimension tables in a star schema.
- The requirement to detect and document complex transformations (e.g., aggregations, type conversions, derived fields) that are necessary for analytical modeling.
- The importance of semantic understanding, as field names and business meanings often differ between operational and analytical models.

This project automates the entire process by:

- **Profiling data** to understand the structure, types, and distributions of each dataset.
- **Generating rich data dictionaries** using LLMs, which provide semantic descriptions and domain values for each field.
- **Leveraging prompt-driven LLM schema matching** to produce detailed, auditable mappings from relational models to star schemas, including all required transformations and semantic justifications.

By focusing on the migration and integration of relational data into star schemas, the project addresses a critical need in modern data engineering and analytics, reducing manual effort and increasing the reliability and transparency of the schema mapping process.

### Required Files and Directory Structure

#### 1. `config.yaml` (root)
Central configuration file. Must define:
  - Paths to all data files, profiles, dictionaries, and models.
  - List of source data dictionaries (`docs/model_sources/`), profiles (`data/profiles/`), samples (`data/samples/`), and full data files (`data/full_files/`).
  - Path to the target star schema model (`docs/model/model.json`).
  - Output directories for LLM results and schema matching.

#### 2. Environment File (`.env` or `.env.example`)
Contains API keys and LLM configuration. Required for LLM-based scripts.

#### 3. Data Files (`data/`)
- `full_files/`: Raw data files for each source (e.g., Parquet, CSV). **Required for sampling and profiling.**
- `samples/`: Sampled data files, generated by `src/samples.py`. **Required for profiling and LLM prompts.**
- `profiles/`: Data profiles, generated by `src/profilling.py`. **Required for dictionary generation.**
- `llm_results/`: Output directory for LLM-generated dictionaries and schema matchings. **Populated by scripts.**
- `distance_calculation/`: Output directory for similarity/comparison results. **Populated by scripts.**

#### 4. Documentation Files (`docs/`)
- `model/`: Must contain the target star schema model as a JSON file (e.g., `model.json`).
- `model_sources/`: Must contain a reference data dictionary (JSON) for each source dataset. These are used as ground truth for evaluation and as input for schema matching.

#### 5. Prompt Templates (`prompts/`)
Contains Markdown files with prompt instructions for LLMs. Required for dictionary and schema matching generation.

### Minimal Setup Checklist

- [ ] `config.yaml` with all paths and lists properly set
- [ ] `.env` or `.env.example` with valid API keys
- [ ] At least one full data file in `data/full_files/`
- [ ] At least one reference data dictionary in `docs/model_sources/`
- [ ] Target star schema model in `docs/model/model.json`
- [ ] Prompt templates in `prompts/`

All other files and directories will be generated by the scripts as you run the pipeline.

`LLM_<TARGET>_*` variables, so you can run several keys, providers, or models in

and Gemini targets can request strict JSON Schema outputs; DeepSeek is kept in
JSON object mode because its OpenAI-compatible API supports JSON output but does
not provide the same strict schema guarantee.

## How It Works

The workflow is modular and each step is performed by a dedicated script in `src/`. Each script can be executed independently or as part of a pipeline, depending on your project needs.

### 1. `src/samples.py` — Sample Generation
Generates random samples of different sizes (100, 1000, 10000) from the full Parquet files defined in `config.yaml`. The samples are saved in `data/samples/` and are used for profiling and dictionary generation.
**Input:** Full Parquet files (`list_full_data_samples` in `config.yaml`).
**Output:** Sampled Parquet files in `data/samples/`.

### 2. `src/profilling.py` — Data Profiling
Uses the `dataprofiler` library to analyze samples and generate detailed profiles (types, statistics, distribution, etc). The profiles are saved as JSON and serve as the basis for dictionary generation.
**Input:** Parquet samples (`list_of_data_samples_profiles` in `config.yaml`).
**Output:** JSON profiles in `data/profiles/`.

### 3. `src/dictionary_generation.py` — Data Dictionary Generation via LLM
Reads the profiles and samples, builds the prompt from the template in `prompts/dictionary_generation.md` and sends it to multiple LLMs (OpenAI, Gemini, DeepSeek). The result is an enriched data dictionary, saved in `data/llm_results/dictionary_llm_results/`.
**Input:** Profiles (`data/profiles/`) and samples (`data/samples/`).
**Output:** LLM-enriched JSON dictionaries in `data/llm_results/dictionary_llm_results/`.


### 4. `src/schema_matching_generation.py` — Schema Matching Generation via LLM
This script is responsible for automating the mapping between relational source models (typically in 3rd Normal Form) and a multidimensional star schema (used in Data Warehouses). The process is guided by a detailed prompt (`prompts/schema_match_generation.md`) that instructs the LLM to:

- **Receive as input:**
  - The source schema (as a JSON data dictionary, possibly enriched with semantic descriptions).
  - The target star schema (as a JSON model).

- **Perform semantic analysis:**
  - Compare field names, types, and descriptions.
  - Leverage synonyms, abbreviations, business context, and PK/FK metadata to infer the best matches.

- **Detect and document transformations:**
  - Identify when data type conversions, aggregations, concatenations, or other transformations are needed to map relational fields to star schema fields.
  - Clearly describe each transformation step in the output.

- **Ensure full coverage:**
  - All target (star schema) fields are included in the mapping, even if no source match is found (in which case, the mapping will indicate no match).
  - Supports 1→N, N→1, and composite/derived mappings, which are common when moving from normalized to denormalized (star) models.

- **Output:**
  - A strict JSON object describing the mapping for each target table and field, including the source(s), semantic description, and any required transformation.
  - The output is saved in `data/schema_matching/` for further analysis or ETL implementation.

This approach is especially focused on the challenges of migrating or integrating data from normalized relational databases to analytical environments based on star schemas, automating what is usually a manual and error-prone process.

### 5. `src/compare_results_dictionary.py` — Dictionary Comparison (Distances)
Compares the dictionaries generated by the LLMs with the reference dictionaries using embeddings and cosine similarity (via `sentence-transformers`). Generates quantitative distance/similarity metrics for evaluation of the results.
**Input:** Reference dictionaries (`docs/model_sources/`) and LLM dictionaries (`data/llm_results/`).
**Output:** Metrics and reports in `data/distance_calculation/`.

### 6. `src/llm.py` — Unified LLM Client
Implements clients for OpenAI, Gemini and DeepSeek, abstracting authentication, prompt sending, response parsing and result saving. Used internally by the dictionary and matching generation scripts.

## Pipeline Orchestrator

`run.py` is the single entry point for the full pipeline. It wraps every step
in a thin subprocess layer, so the existing `src/*.py` scripts keep their
direct CLI invocations and tests stay decoupled. The orchestrator exposes the
following subcommands:

| Subcommand   | Wraps                              | Purpose                                                                                      |
| ------------ | ---------------------------------- | -------------------------------------------------------------------------------------------- |
| `bootstrap`  | `src/bootstrap_bird_mini_dev.py`   | Download/extract BIRD Mini-Dev and generate dictionaries, samples, profiles, and the manifest. |
| `generate`   | `src/dictionary_generation.py`     | Run the LLM dictionary generation for every configured profile.                              |
| `validate`   | (in-process audit)                 | Inspect every `*_parsed.json` under `data_llm_results_dictionary_generation.path` for errors. |
| `retry-llm`  | `src/dictionary_generation.py`     | Rerun only the provider/profile combinations whose parsed JSON is missing or invalid.        |
| `compare`    | `src/compare_results_dictionary.py` | Compute cosine-similarity distances and write per-table and per-model summaries.            |
| `all`        | validate → (retry-llm) → compare   | Convenience wrapper for the post-generation steps.                                           |
| `pipeline`   | bootstrap → generate → all         | One-shot end-to-end pipeline with optional flags.                                           |

Quick examples:

```bash
# Bootstrap the BIRD Mini-Dev benchmark and refresh config.yaml in one shot
python run.py bootstrap --update-config

# Generate LLM dictionaries for every profile
python run.py generate

# Audit the LLM results without calling any API
python run.py validate

# Rerun only the failed provider/profile combinations
python run.py retry-llm

# Compute similarity scores against the reference dictionaries
python run.py compare

# Full pipeline (bootstrap + generate + validate + retry + compare)
python run.py pipeline --with-bootstrap --with-generate --with-retry
```

`run.py` exits with the last non-zero return code from the wrapped script, so
it composes naturally with `set -e` and CI pipelines.

## Example Usage
### 1. Install Dependencies

This project recommends using [uv](https://github.com/astral-sh/uv), an ultra-fast Python package manager, for dependency management and reproducible environments. `uv sync` ensures your environment matches the dependencies specified in `pyproject.toml` and `poetry.lock` (if present).

**Install uv (if not already installed):**
```bash
pip install uv
# or, with pipx (recommended):
pipx install uv
```

**Sync dependencies:**
```bash
uv sync
```

This will create a virtual environment (if one does not exist) and install all required packages. You can continue to use `uv` for fast installs and updates, or use `poetry`/`pip` as needed for compatibility.

### 2. Configure Environment

- Copie `.env.example` para `.env` e preencha com suas chaves de API e configurações de modelo.
- Ajuste o `config.yaml` conforme necessário para os caminhos dos seus dados e preferências.

### 3. Gerar Amostras dos Dados

Para gerar amostras dos arquivos Parquet completos, execute:
```bash
python src/samples.py
```
Isso irá criar amostras de tamanhos 100, 1000 e 10000 para cada arquivo listado em `list_full_data_samples` no `config.yaml`, salvando em `data/samples/`.

### 4. Executar o Data Profiling
```bash
python src/profilling.py
```

### 5. Gerar Dicionários de Dados
```bash
python src/dictionary_generation.py
```

### 6. Executar o Schema Matching
```bash
python src/schema_matching_generation.py
```

### 7. Comparar Distâncias dos Dicionários
```bash
python src/compare_results_dictionary.py
```
Esse script compara apenas as distâncias entre os dicionários gerados, útil para avaliação quantitativa dos resultados de matching.

### 8. Bootstrap do BIRD Mini-Dev
Para baixar/processar o benchmark BIRD Mini-Dev e gerar dicionários-base, amostras e perfis compatíveis com o pipeline existente:
```bash
uv run python src/bootstrap_bird_mini_dev.py --update-config
```

O script consome o ZIP oficial do Mini-Dev quando necessário, localiza todos os bancos SQLite em `dev_databases`, lê os CSVs de `database_description`, introspecta schemas SQLite e gera:

- Dicionários-base em `data/bird_mini_dev/dictionaries/`
- Samples de prompt em `data/bird_mini_dev/samples/`
- Samples maiores para profiling em `data/bird_mini_dev/profile_samples/`
- Perfis compactos em `data/bird_mini_dev/profiles/`
- Manifest em `data/bird_mini_dev/manifest.json`

Com `--update-config`, as listas `list_of_profiles`, `list_of_data_samples`, `list_of_data_samples_profiles` e `list_of_data_dictionaries` passam a apontar para os artefatos BIRD gerados, permitindo executar `src/dictionary_generation.py` e `src/compare_results_dictionary.py` sem adaptar seus contratos.

### 9. Orquestração via `run.py`
Todos os passos acima (incluindo o bootstrap do BIRD Mini-Dev) podem ser executados a partir do orquestrador `run.py`. Veja a seção [Pipeline Orchestrator](#pipeline-orchestrator) para a lista completa de subcomandos. Em resumo:
```bash
# Pipeline completo: bootstrap + generate + validate + retry-llm + compare
uv run python run.py pipeline --with-bootstrap --with-generate --with-retry

# Apenas validar, opcionalmente rerodar erros, e comparar
uv run python run.py all --with-retry
```

## Prompt Templates

Prompt templates for LLMs are stored in the `prompts/` directory. These templates define exactly how the LLMs are instructed to generate data dictionaries and schema matchings, ensuring consistency and high-quality outputs.

### How Prompts Are Used

- **dictionary_generation.md**: Used by the script `src/dictionary_generation.py` to instruct the LLM to generate a complete data dictionary for a dataset. The prompt provides:
  - The context (profiling statistics and a sample of the data).
  - Clear instructions for the LLM to output a JSON with table and field descriptions, types, example values, and possible domain values.
  - The LLM receives two blocks: `## PROFILE` (profiling info) and `## SAMPLE` (data sample), and must return only a valid JSON as specified in the prompt.

- **schema_match_generation.md**: Used by the script `src/schema_matching_generation.py` to instruct the LLM to generate a mapping between source and target schemas. The prompt provides:
  - Two JSON blocks: one for the source schema and one for the target schema.
  - Step-by-step instructions for the LLM to compare, analyze, and map fields, including semantic analysis and possible transformations.
  - The LLM must return only a valid JSON object with the mappings, following the strict structure described in the prompt (no extra text or comments).

#### Customizing Prompts

You can edit the prompt files in `prompts/` to:
- Refine instructions for your specific data domain or business rules.
- Change the output format or add constraints for the LLM.
- Improve clarity or add examples to guide the LLM for better results.

**Tip:** Always ensure the output format in the prompt matches what your scripts expect, as the scripts parse the LLM output directly as JSON.

## Roadmap & Future Work

The following directions are suggested for future development and research using this framework:

### 1. Enhanced Analysis and Validation of Schema Matching Results
- Develop more robust evaluation metrics and visualization tools to assess the quality of the generated schema mappings.
- Implement manual and semi-automated validation workflows, allowing domain experts to review, approve, or correct LLM-generated mappings.
- Integrate feedback loops where corrections can be used to fine-tune prompts or LLM behavior.

### 2. Experimentation with Alternative Prompts
- Design and test new prompt templates for both data dictionary generation and schema matching, aiming to improve semantic accuracy and coverage.
- Explore prompt engineering techniques such as few-shot examples, chain-of-thought, and explicit transformation instructions.
- Compare the performance of different prompt strategies across various LLM providers.

### 3. Broader Dataset Support
- Extend the framework to support additional public and private datasets beyond NYC Taxi, including healthcare, finance, retail, and IoT domains.
- Develop adapters for different data formats and schemas, ensuring the pipeline is robust to diverse data sources.
- Benchmark the system’s performance and generalizability across these new datasets.

### 4. Multi-Agent Module for Automated Code Generation
- Design a multi-agent orchestration module that leverages LLMs to automatically generate integration code in SQL, Python, and PySpark.
- Enable the system to produce ETL scripts, transformation logic, and data loading routines directly from the schema mappings.
- Support code validation, testing, and deployment workflows, making the transition from schema mapping to operational data integration seamless.

These enhancements will further automate, generalize, and operationalize the schema matching and data integration process, making the framework even more valuable for real-world data engineering and analytics projects.
